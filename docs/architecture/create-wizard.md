# イベント作成ウィザード設計メモ

## 背景

`/create` は1画面に多くの入力が集まり、初見で「何から手を付ければよいか」が分かりづらい課題があった。
入力手順を段階化し、迷いを減らしつつ必要情報の入力を確実にするためにウィザード化を行う。

## 目的

- 入力の順序を明確にし、理解と完了率を向上させる。
- 方式選択と設定入力を分離し、意思決定の負荷を下げる。
- 既存のサーバーアクションやDBスキーマに影響を与えずUIだけで改善する。

## 構成

### ステップ1: 基本情報

- イベントタイトル（必須）
- 説明（任意）
- タイトル未入力の場合は次へ進めない

### ステップ2-1: 入力方式選択

- 自動作成 / 手動選択のカード型ラジオで提示
- 方式ごとに「向いている人」「できること」「所要感」を短文で提示
- 方式未選択では次へ進めない

### ステップ2-2: 設定

- 開始日 / 終了日
- 時間枠の長さ
- 各日の開始時刻 / 終了時刻
- 自動方式: 期間と時間帯から候補枠を自動生成
- 手動方式: 期間と時間帯の設定後にカレンダー選択へ進む

### ステップ2-3: カレンダー選択（手動方式のみ）

- 期間・時間帯から生成されたカレンダー表でドラッグ選択
- 候補枠が0の場合は次へ進めない

### ステップ3: 確認・作成

- 入力内容の要約を表示（タイトル/説明/方式/日付範囲/時間帯）
- 利用規約同意をここに集約
- 作成ボタンで `createEvent` を実行

## 方式選択をサブステップ化する理由

- 方式選択は「作成の方針」を決める重要な意思決定であり、他入力と混在させると認知負荷が高い。
- 先に方式の理解と選択を完了させることで、後続の設定項目が「なぜ必要か」を理解しやすくなる。

## バリデーション方針

- ステップごとに入力必須条件を判定し、未達の場合は進行をブロックする。
- 作成時には全ステップの検証を再実行する。
- 既存のサーバーサイド検証は維持し、フロント側はユーザー体験向上を目的とする。

## 補足（UI/挙動）

- 日付・時間帯・時間枠の入力は横並びで簡潔に配置し、モバイルでも視認性を確保する。
- ステップ遷移時にステップ見出しへスクロールするが、初回表示はページ認知を優先してスクロールしない。

## 影響範囲

- 変更は `EventFormClient` を中心としたUIに限定する。
- `createEvent` のI/OやDBスキーマは変更しない。
- E2E/ユニットテストは新しい導線に合わせて更新する。
