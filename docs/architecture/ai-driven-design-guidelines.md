# AI駆動開発におけるデザイン品質ガイド（2026-03-12）

## 背景

- AI を使った UI の初稿生成は速度面で有効だが、要件や文脈を十分に与えないと「どのサービスにも見える」「既視感が強い」画面になりやすい。
- GOV.UK のデザイン原則では、まずユーザーニーズを起点にし、思い込みではなく調査・データ・対話から設計することを重視している。
- Figma も AI 出力は出発点であり、最終成果物は人の判断とクラフトで磨く前提を明示している。
- そのため本プロジェクトでは、AI を「完成品メーカー」ではなく「探索を速める補助」として扱い、調査・文脈化・レビューを必須工程にする。

## このガイドの結論

1. **検索と調査を先に行い、AI には要件整理済みの文脈だけを渡す。**
2. **AI の初稿をそのまま採用しない。必ず情報設計・文言・トークン・余白・動きの再設計を行う。**
3. **ブランド、ユーザー文脈、利用シーンに結びつかない装飾は採用しない。**
4. **配色・書体・余白・モーションはトークン化し、場当たりな値を増やさない。**
5. **アクセシビリティと可読性を優先し、見た目の派手さを優先しない。**

## AI感が強くなる典型パターン

以下は外部資料と本リポジトリの現状を踏まえた、実務上のアンチパターンである。

- ユーザー課題ではなく「それっぽい LP 構成」から作り始める。
- `Hero + 特徴3列 + FAQ + CTA` のようなテンプレ構成を、根拠なく毎回流用する。
- 目的の説明がないまま、紫系グラデーション、ガラス風カード、過剰な発光、丸すぎるバッジを重ねる。
- 書体・余白・色・角丸・影の根拠がなく、画面単位で値が増殖する。
- 実ユーザーの言葉ではなく、抽象的で汎用的なコピーを置く。
- 情報伝達に不要なアニメーションを足し、`prefers-reduced-motion` を考慮しない。
- 既存のサービス文脈や機能差ではなく、AI が学習しやすい一般的 SaaS 見た目に寄せる。

## 具体的に避けるべきデザイン例

レビュー時に「これは AI 感が強いか」を判定しやすいよう、見た目の兆候ごとに具体例を列挙する。

### 1. 汎用 SaaS ヒーローの量産

- 画面上部が毎回 `大きい見出し + 抽象的サブコピー + バッジ3個 + CTA2個 + 右側イラスト` になっている。
- 見出しが `スケジュールをもっとスマートに` `チームの生産性を最大化` のように、何のサービスにも当てはまる。
- ヒーローの右側に、実機能と関係の薄いダッシュボード風モックや浮遊カードを置いている。

避ける理由:

- 「何ができるか」より「AI が生成しやすい構成」が前面に出る。
- 初回訪問者にとって必要な理解より、見慣れた雰囲気が優先される。

### 2. 紫グラデーション依存

- 背景、CTA、バッジ、見出し強調、アイコン発光がすべて青紫からピンク系のグラデーションで統一されている。
- 主色の理由がなく、とりあえず `from-indigo-500 via-violet-500 to-fuchsia-500` のような値を置いている。
- 白背景に淡い紫テキストを重ね、コントラストが不足している。

避ける理由:

- 「AI が出しがちな見た目」に直結しやすく、ブランド差が消える。
- 色の意味づけが弱く、状態表現や情報優先度も崩れやすい。

### 3. ガラス風カードの多用

- 主要カードすべてに `backdrop-blur`, 半透明白, 細い白枠, 大きい影を載せている。
- 背景にグラデーションや光彩を置いたうえで、前景カードも半透明にしている。
- 実際には情報量が少ないのに、見た目だけ豪華な「浮いたカード」が増えている。

避ける理由:

- 情報密度が低いまま派手さだけが増える。
- 可読性、コントラスト、モバイル時の見やすさを損ねやすい。

### 4. バッジの過剰装飾

- ヒーロー直下に `無料` `高速` `かんたん` `安心` `AI搭載` のような短いバッジを大量に並べる。
- バッジ自体は多いのに、本文では機能差や価値が説明されていない。
- バッジの色が多すぎて、どれが主情報か分からない。

避ける理由:

- 説明を省略して雰囲気だけで価値を伝えようとしている状態になりやすい。
- 画面の密度を上げるわりに理解は進まない。

### 5. アイコン依存の特徴セクション

- `特徴 3-6 個` がすべて「丸背景のアイコン + 1 行タイトル + 1 行説明」で並ぶ。
- アイコンを消すと、情報の差がほぼなくなる。
- すべて同じボックス・同じ情報量・同じ視覚強度で並んでいる。

避ける理由:

- 何が重要かの序列が消え、ただのテンプレ構成に見える。
- サービス固有の強みより、均一なカード UI の印象が勝つ。

### 6. ダミーの統計表示

- `10,000+ Users` `99.9% Uptime` `4.9 Rating` のような数値を根拠なく置く。
- 実サービスにない導入実績や満足度風の数字を、装飾として並べる。
- 実データがなくても、見栄え目的で統計セクションを入れる。

避ける理由:

- 信頼を損なう。
- AI テンプレ感が非常に強く、内容の薄さが目立つ。

### 7. コピーが抽象的すぎる

- `未来の働き方へ` `次世代の体験` `シームレスなコラボレーション` のように、名詞が抽象語ばかりになっている。
- ユーザーの行動が書かれておらず、操作後に何が起こるか分からない。
- 誰向けか、どの状況向けか、何が短縮されるかが本文から読み取れない。

避ける理由:

- 読んでも使い方が想像できず、印象語だけが残る。
- 実サービスの価値ではなく、マーケティングテンプレの匂いが強くなる。

### 8. 過剰な背景装飾

- 円形ブラー、ノイズ、格子、星、浮遊図形、線画、発光などを複数重ねる。
- 背景の主張が強く、本文や CTA より目立っている。
- スマホ幅で背景装飾が切れて、意味のない余白やにじみが発生する。

避ける理由:

- 画面の目的より演出が前に出る。
- 実用画面では特に、集中を妨げやすい。

### 9. 角丸と影の盛りすぎ

- すべてのカード、ボタン、入力欄が極端に丸い。
- 影が深く、複数レイヤーで重なっている。
- 角丸も影も強いのに、レイアウトや情報構造には意味がない。

避ける理由:

- 「丸くして影を付ければ今風」という AI 的近道になりやすい。
- 触れるべき要素と単なる箱の差が曖昧になる。

### 10. モーションの演出過多

- 要素すべてが `fade-up` `scale-in` `stagger` で順番に出る。
- スクロールするたびにカードがふわっと浮く、揺れる、拡大する。
- 情報理解よりも「動いていること」自体が主役になっている。

避ける理由:

- 繰り返し操作時にノイズとなる。
- reduced motion の観点でも不適切になりやすい。

### 11. フォームが装飾優先

- 入力欄が光る、揺れる、背景が変化するなど、常時エフェクトが入っている。
- 必須項目や次の操作より、見た目の飾りが目立っている。
- プレースホルダーが説明文を兼ねており、ラベルが弱いか存在しない。

避ける理由:

- 実用タスクの成功率を下げる。
- 入力体験で必要なのは華やかさより、明確さと復帰しやすさである。

### 12. 本文より先に「世界観」を語る

- サービスの具体機能説明より先に、長いコンセプト文やブランドポエムを置く。
- `私たちは人と時間を再定義します` のような表現が先に来る。
- 使い始める方法が、数スクロールしないと見つからない。

避ける理由:

- ユーザーは思想より先に「自分に使えるか」を判断したい。
- 実タスク導線が後退し、LP テンプレ感が強まる。

## 避けるべき実装断片の例

以下のような断片は、理由が明確でない限り採用しない。

- `backdrop-blur-xl bg-white/10 border border-white/20`
- `bg-gradient-to-r from-indigo-500 via-purple-500 to-pink-500`
- `shadow-[0_20px_80px_rgba(80,70,180,0.35)]`
- `rounded-full px-6 py-3` を情報性の低いバッジへ多用
- すべてのセクションに `whileInView` アニメーションを付与
- 価値説明のない `badge badge-primary`, `badge badge-secondary` の連続配置

## 「AIっぽい」を見抜く簡易質問

- その装飾を外しても、価値提案は伝わるか。
- その色や影や角丸に、サービス固有の理由があるか。
- そのセクションは、実データか実機能か実ユースケースに結びついているか。
- その文言は、他の SaaS 名に置き換えても成立してしまわないか。
- そのアニメーションは、無効化した時に困る人がいるほど必要か。

## 事前調査の必須プロセス

デザイン着手前に、最低でも次の 4 種類の情報を集める。

### 1. ユーザーと利用文脈

- 対象ユーザーは誰か。
- どの場面で、何分以内に、何を終えたいのか。
- 迷いやすい操作は何か。
- モバイル利用か、PC 利用か、両方か。

最低限まとめる項目:

- 主要ユーザー 1-2 種
- 主タスク 1 つ
- 補助タスク 2-3 つ
- 失敗すると困る点
- 使われるデバイス・時間帯・場所

### 2. 競合・類似サービスの比較

- 同種サービス 2-3 件
- 近い操作体験を持つ別カテゴリサービス 2 件前後
- 良い点と悪い点をスクリーンショットではなく「構造」と「判断理由」で比較する

確認観点:

- 情報の優先順位
- CTA の置き方
- フォームの深さ
- モバイル時の崩れ方
- 独自性が出ている要素

### 3. 一次情報ベースのデザイン原則

- 公式デザインシステム
- 公式アクセシビリティ基準
- 使う UI ライブラリ / フレームワークの制約

本プロジェクトで最低限確認する観点:

- 可読性
- 一貫性
- トークン運用
- 動きの必然性
- reduced motion 対応

### 4. 既存実装との整合

- `src/app/globals.css` のトークン・基礎スタイル
- 既存ページの見出し階層
- 既存コンポーネントの角丸・影・余白・配色
- すでに導入しているアニメーション方針

## AI に渡す前に作る「デザインブリーフ」

AI へ直接「良い感じに作って」と投げない。先に次の形式で要件を固定する。

```md
### 目的
- この画面で最優先に達成したいことは何か

### ユーザー
- 誰が使うか
- どの状況で使うか

### UI の優先順位
- 最初に見せる情報
- 2 番目に見せる情報
- 押してほしい行動

### このサービスらしさ
- 機能上の強み
- 言葉遣いのトーン
- 避けたい印象

### 実装制約
- 既存トークンを優先する
- Server Components 優先
- モバイル/PC 両対応
- prefers-reduced-motion を尊重する

### 禁止事項
- 汎用 SaaS 風のヒーロー
- 根拠のない紫グラデーション常用
- ガラス風 UI の多用
- 装飾だけのモーション
- 内容の薄いバッジ列
```

## AI への依頼ルール

### 依頼時に必ず含めること

- ユーザー像
- 主タスク
- 既存画面と揃えるべき点
- あえて変えてよい点
- 使ってよい色数と書体の方針
- 避けたい表現

### 依頼文で明示すること

- 「3案出す」よりも「1案に絞って根拠を添える」
- 「見た目」より「情報の優先順位」を先に説明させる
- 「なぜこの色・余白・書体・動きなのか」を言語化させる
- 「このサービス特有の文脈に置き換えた文言」を出させる

### 依頼文の例

```md
日程調整アプリのトップ画面を改善したい。
目的は「初回訪問ユーザーが、ログイン不要で使えることを3秒以内に理解し、イベント作成を始められること」。

以下を守ること。
- 一般的な SaaS LP のテンプレ構成をそのまま使わない
- 既存の情報設計を尊重しつつ、主 CTA までの理解コストを下げる
- 色・書体・余白・モーションの採用理由を各 1 文で説明する
- モバイルでの見え方を優先して構成する
- 装飾目的のグラデーション、ガラス風 UI、過剰なバッジ列は使わない
- このサービスの文脈に合う見出し・本文を日本語で書く
```

## 実装時のガードレール

### 1. 情報設計を先に決める

- まずワイヤー相当のテキスト構造を作る。
- 見出し、本文、補助情報、CTA の順序が固まるまで色や装飾を足さない。
- 1 セクション 1 メッセージを守る。情報を詰め込みすぎない。

### 2. 文字組みで差を作る

- 書体は「デフォルトだから」ではなく、ブランド表現か可読性かの目的で選ぶ。
- 同一画面で見出し・本文・補足の階層差を明確にする。
- マーケティング面では、既定の `Inter + 太字 + 紫アクセント` に安易に寄せない。
- 長文では行間と段落間隔を優先し、圧縮して詰め込まない。

### 3. 色は意味で使う

- 主色・補助色・状態色の役割を分ける。
- 強調のたびに新色を足さない。
- コントラスト不足の淡色テキストや、背景依存の読みにくいグラデーション文字は避ける。

### 4. 余白とレイアウトで「それっぽさ」を消す

- 余白はトークン化し、気分で `px-7` や `gap-[18px]` を増やさない。
- レイアウトはコンテンツ幅、行長、視線移動を理由に決める。
- 「何となくカードを並べる」構成を避け、比較・手順・実績など意味のある並びにする。

### 5. モーションは意味がある時だけ使う

- モーションは状態変化の理解補助に限定する。
- ローディング、モーダル開閉、保存完了など文脈がある場所を優先する。
- スクロール連動の装飾、不要なパララックス、過剰な拡大縮小は避ける。
- `prefers-reduced-motion` を前提に設計する。

### 6. AI 初稿から必ず人が崩す

- 見出し文言を実ユーザー向けに言い換える。
- 既視感の強いセクションを削る。
- 1 つ以上、サービス固有の説明要素を前段へ持ってくる。
- アイコン、背景装飾、バッジの数を半分以下に減らせないか確認する。

## レビュー時のチェックリスト

- その画面は、サービス名を隠しても他サービスと見分けがつくか。
- 主 CTA の前に、ユーザーが理解すべき文脈が整理されているか。
- 余白、書体、色、角丸、影の値に再利用可能なルールがあるか。
- 追加したモーションは、無効化しても価値が残るか。
- UI の説明文が、実際のユースケースに根差しているか。
- モバイル幅で、見出しと CTA が最初の視界に収まるか。
- 「雰囲気」ではなく、ユーザー行動の改善仮説を説明できるか。

## 本リポジトリでの運用ルール

### ドキュメント

- デザイン方針を変える場合は `docs/architecture` に背景・理由・影響範囲を残す。
- README のドキュメント一覧を更新する。
- 破壊的変更や運用ルール追加は `docs/CHANGELOG.md` に追記する。

### 実装

- 既存の DaisyUI / Tailwind 利用時も、デフォルト見た目をそのまま採用せず意図を上書きする。
- 新規 UI では、ハードコードした色・余白・影を増やす前に共通トークン化を検討する。
- Server Components 優先方針を維持し、視覚表現のためだけに不要な Client Component 化をしない。

### レビュー

- 「AI っぽい」かどうかは感覚ではなく、上記チェックリストで指摘する。
- 指摘時は「どのユーザーニーズに対して不要か」「どの一貫性を崩しているか」まで書く。

## 参考資料から本ガイドへ落とした判断

- GOV.UK の「Start with user needs」「Do the hard work to make it simple」「Be consistent, not uniform」を、AI への依頼前にブリーフを固定する運用へ変換した。
- Atlassian の typography / foundations / tokens を踏まえ、書体・色・余白・動きを単発値ではなく設計判断として扱う方針にした。
- Apple と W3C の reduced motion 指針を踏まえ、非必須モーションの削減と設定尊重を必須にした。
- Figma の AI 関連資料を踏まえ、AI 出力は初稿に留め、デザインシステムの文脈を与えない生成は採用しない方針にした。
