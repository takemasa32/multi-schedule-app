# Motion 導入方針（低影響・段階移行）

## 背景

- 現在はランディング系コンポーネントで `framer-motion` を利用しており、体感品質（表示の滑らかさ・意図伝達）を担保している。
- Motion 公式（motion.dev）では React 向けパッケージとして `motion` が提供されており、既存 API に近い記法で移行できる。
- 全画面を一気に作り替えると回帰リスクが高いため、既存挙動を維持しつつ依存パッケージを先に置き換える。

## 今回の方針（Phase 1）

1. 依存を `framer-motion` から `motion` へ置換する。
2. 既存の `motion` コンポーネント記法（`motion.div` 等）は維持し、import のみ `motion/react` へ変更する。
3. まずは既存でアニメーションを使っている画面（ランディング）に限定して差分を適用する。

> これにより、UX 低下リスクを抑えながら、今後の全画面最適化に向けた土台を先に揃える。

## 各画面の体感最適化ガイド（Phase 2 以降）

- **一覧系画面**: 初回表示は `opacity + y` の短い遷移（120-180ms）で認知負荷を下げる。
- **フォーム系画面**: エラー表示・保存完了のみ強調し、常時アニメーションは避ける。
- **モーダル/ドロワー**: `AnimatePresence` と `exit` を必須化し、閉じる時の文脈喪失を防ぐ。
- **ローディング状態**: スケルトン優先、無限ループ系アニメーションは最小限にする。
- **低性能端末配慮**: `prefers-reduced-motion` を尊重し、移動量を減らす。

## 実装ルール

- 画面ごとに「目的のある遷移」だけを採用し、装飾目的のアニメーションを増やしすぎない。
- 体感遅延を避けるため、遷移時間は原則 200ms 以下を優先する。
- Server Components 優先方針を維持し、アニメーションが必要な範囲だけ Client Component 化する。

## 影響範囲

- 依存ライブラリ: `framer-motion` -> `motion`
- 変更ファイル: ランディング UI 2 ファイルの import 行のみ
- 挙動互換性: 既存 variant / transition 設定はそのまま利用

## ロールアウト案

1. Phase 1（今回）: 依存置換 + 既存利用箇所の import 差し替え。
2. Phase 2: `/account` とイベント作成/回答導線の「操作結果フィードバック」重点改善。
3. Phase 3: 共通アニメーショントークン（duration/easing）を定義し、全画面で統一。
