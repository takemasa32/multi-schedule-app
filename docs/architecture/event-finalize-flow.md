# イベント日程確定フロー設計メモ

作成日: 2026-03-11  
最終更新日: 2026-03-11

## 背景

- イベント詳細内の折りたたみ UI では、主催者が「いま何を変更するのか」を把握しづらかった。
- 確定は影響範囲が大きい操作のため、候補選択と最終確認を分けた専用導線が必要だった。
- 回答入力や同期確認と同様に、タスク単位で専用ルートへ遷移する設計へ揃えたい。

## 方針

- イベント詳細ページは「現在の確定状況の把握」と「確定画面への遷移」に役割を限定する。
- 実際の確定操作は `/event/{public_id}/finalize` に集約する。
- 専用ルートでは 2 ステップ構成とし、候補選択と確認を分離する。

## 画面構成

### 1. イベント詳細 `/event/{public_id}`

- 「日程の確定」セクションは概要カードとして表示する。
- 表示内容:
  - 現在の確定件数
  - 確定済み日程一覧（存在する場合）
  - 専用ルートへの CTA
- 補助説明は最小限にし、状態と操作に必要な情報だけを出す。
- 候補マトリクスや即時確定ボタンは置かない。

### 2. 日程確定 `/event/{public_id}/finalize`

- 上部にパンくず、タイトル、進捗表示を設置する。
- Step1: 候補を選ぶ
  - 候補マトリクスで日程を選択/解除
  - 選択件数、全選択で参加可能な人数、現在の確定内容を表示
  - モバイルでは下部固定アクションで「内容を確認する」を維持
  - モバイルではセルサイズと余白を圧縮し、表と CTA を優先して見せる
- Step2: 内容を確認
  - 新しく確定する枠
  - 解除される枠
  - 維持される枠
  - 全選択で参加可能な人

## 確認・完了

- 実行直前の最終確認は共通 `ConfirmationModal` を使う。
- `0件選択 + 既存確定あり` は「全解除」として扱う。
- 完了後は `/event/{public_id}?finalize_status=saved|cleared` に戻し、結果バナーを表示する。
- バナー表示後はクエリを自動的に除去し、再読み込み時に残さない。

## 非目標

- 権限制御の追加
- 確定ロジック自体の DB 仕様変更
- 回答集計 UI の再設計
